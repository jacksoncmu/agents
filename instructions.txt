Tool: summarize_columns
Description:

Required input object:
{
  "dataset_id": string,
  "columns": string[],
  "include_quantiles": boolean
}

Rules:
- columns must be an array of exact column names

- include_quantiles must be true or false, never omitted

Returns:
{

Tool: split_dataset
Description:
Split one dataset into train/validation/test partitions for supervised learning.

Required input object:
{
  "dataset_id": string,
  "target_column": string,
  "task_type": "classification" | "regression",
  "train_fraction": number,
  "validation_fraction": number,
  "test_fraction": number,
  "random_seed": integer,
  "stratify": boolean
}

Rules:
- train_fraction + validation_fraction + test_fraction must equal 1.0
- stratify may only be true for classification
- target_column must be a single string, not an array
- do not call this tool until the target column is explicitly identified
- fail if target_column is missing from dataset schema

Returns:
{
  "split_id": string,
  "dataset_id": string,
  "task_type": "classification" | "regression",
  "target_column": string,
  "partitions": {
    "train_rows": integer,
    "validation_rows": integer,
    "test_rows": integer
  }
}

Tool: train_model
Description:
Train one baseline model on an existing supervised split.

Required input object:
{
  "split_id": string,
  "model_family": "linear_regression" | "logistic_regression" | "random_forest" | "xgboost" | "lightgbm",
  "feature_columns": string[],
  "target_column": string,
  "hyperparameters": object
}

Rules:
- split_id must come from split_dataset
- feature_columns must be an array of exact column names
- target_column must exactly match the split target
- do not include the target column inside feature_columns
- hyperparameters must be an object, use {} if defaults are intended
- reject training if leakage checks fail
- reject unsupported model_family values

Returns:
{
  "model_id": string,
  "model_family": string,
  "task_type": "classification" | "regression",
  "feature_columns": string[],
  "target_column": string,
  "training_summary": {
    "train_rows": integer,
    "validation_rows": integer,
    "fit_status": "success" | "failed"
  }
}

Tool: evaluate_model
Description:
Evaluate one trained model on validation or test data.

Required input object:
{
  "model_id": string,
  "evaluation_split": "validation" | "test",
  "metrics": string[]
}

Rules:
- metrics must be an array of metric names
- allowed metrics depend on task_type
- for classification, allowed metrics:
  ["accuracy", "precision", "recall", "f1", "roc_auc", "log_loss"]
- for regression, allowed metrics:
  ["mae", "rmse", "r2", "mape"]
- reject incompatible metrics

Returns:
{
  "model_id": string,
  "evaluation_split": "validation" | "test",
  "metrics": {
    "<metric_name>": number
  }
}

5. Keep system-layer, scenario-layer, and execution-layer prompts/config separate

Implement three layers clearly:

System layer:
- role and capability boundary
- tool-use discipline
- message protocol constraints
- honesty about uncertainty
- never fabricate results

Scenario layer:
- ML workflow rules
- EDA-before-modeling preference
- leakage prevention
- task-type-specific metric discipline
- allowed model families
- artifact/report expectations

Execution layer:
- reply in the user’s language
- when giving findings, include concrete numbers
- when recommending a model, include why
- when evaluating a model, report metric values explicitly
- when analysis is incomplete, say what is complete vs pending
- when ambiguity exists, state the assumption used
- output format for reports, tables, and saved artifacts

Do not mix these responsibilities together.
